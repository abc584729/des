# DES 加密收发链路改造说明

## 1. 项目目标

在保持现有 AD/DA、GT、PAM4 调制及训练同步链路不变的前提下，将原 LSTM 混沌加密替换为 DES 加密，完成发送端加密、接收端解密及端到端数据正确性验证。

本次改造遵循以下原则：

- 复用已经调通的 AD/DA 接口和 GT lane 对齐流程。
- 复用现有训练序列、互相关同步、采样点调整及 PAM4 判决框架。
- 保持现有物理层数据格式不变。
- 分阶段验证，先确认无加密链路，再接入 DES，便于定位问题。

## 2. 技术方案

### 2.1 物理层参数

| 参数 | 设计值 |
| --- | ---: |
| 用户时钟 | 120 MHz |
| 每时钟 PAM4 符号数 | 16 symbols |
| 每符号采样数 | 8 SPS |
| 每时钟采样点数 | 128 samples |
| 单采样点宽度 | 6 bit |
| AD/DA 并行数据宽度 | 768 bit |
| 每时钟逻辑数据量 | 32 bit |

一个120 MHz时钟传输16个 PAM4 符号，每个符号承载2 bit，因此每时钟承载32 bit逻辑数据：

```text
16 symbols × 2 bit/symbol = 32 bit/clock
```

经过8倍采样后，每时钟形成完整的768 bit AD/DA采样帧：

```text
16 symbols × 8 SPS × 6 bit/sample = 768 bit/clock
```

### 2.2 DES 数据适配

DES 每次处理一个64 bit数据块，而现有PAM4链路每时钟承载32 bit，因此一个DES密文块分两个120 MHz时钟发送：

```text
第1拍：cipher[31:0]  → 16个PAM4符号
第2拍：cipher[63:32] → 16个PAM4符号
```

接收端按相同顺序恢复：

```text
第1拍恢复 cipher[31:0]
第2拍恢复 cipher[63:32]
→ 拼接为64 bit密文
→ DES解密
→ 输出64 bit明文
```

### 2.3 发送端数据流

```text
64 bit明文
→ DES加密
→ 64 bit密文
→ 64转2×32 bit
→ 每拍映射为16个PAM4符号
→ 复用原8倍脉冲成形
→ 768 bit DAC数据
→ 原AD/DA及GT发送链路
```

### 2.4 接收端数据流

```text
原AD/DA及GT接收链路
→ 768 bit ADC数据
→ 训练同步、采样点调整及LMS均衡
→ 每拍判决16个PAM4符号
→ PAM4解映射得到32 bit
→ 连续两拍拼接64 bit密文
→ DES解密
→ 明文比较、CRC或错误计数
```

## 3. 同步方案

业务同步继续复用现有流程：

```text
发送端循环发送训练序列
→ 接收端检测信号幅度变化
→ 128点互相关确定采样窗口
→ 完成训练序列对齐
→ 发送端在窗口边界切换到有效数据
→ 接收端检测训练数据到有效数据的变化
```

DES模式下，接收端检测到有效数据起点后：

1. 清零32/64 bit拼接状态。
2. 将第一个有效窗口认定为密文低32 bit。
3. 将第二个有效窗口认定为密文高32 bit。
4. 每两个有效窗口组成一个DES密文块。

第一阶段不增加周期同步头，优先复用当前训练同步。后续可根据失锁恢复需求增加固定帧头、帧序号和CRC。

## 4. 调试计划

### 4.1 阶段一：无加密PAM4链路验证

目的：确认去除混沌加解密后，现有板间物理链路仍能正确传输和恢复PAM4数据。

工作内容：

- 旁路发送端混沌叠加模块。
- 旁路接收端本地LSTM和混沌相减模块。
- 保留原训练同步、8倍脉冲成形、采样点调整、LMS和PAM4判决。
- 发送固定PAM4测试ROM。
- 使用接收端相同参考ROM统计SER。

阶段验收条件：

- 训练同步能够稳定完成。
- 接收端PAM4判决结果与发送测试序列一致。
- SER达到双方约定指标。
- 连续运行过程中无明显失锁或数据错位。

### 4.2 阶段二：DES端到端链路验证

目的：在阶段一链路稳定的基础上接入DES加解密模块。

工作内容：

- 发送端接入DES加密和64/32 bit适配模块。
- 接收端接入32/64 bit拼接和DES解密模块。
- 第一版使用收发端相同的固定DES密钥。
- 使用标准DES测试向量验证算法正确性。
- 使用固定明文ROM完成板间端到端测试。
- 统计密文恢复错误、解密明文错误和连续运行状态。

阶段验收条件：

- DES标准测试向量结果正确。
- 接收端64 bit密文分组边界正确。
- 解密明文与发送明文逐块一致。
- 训练完成后能够连续稳定传输。
- 错误计数或CRC结果达到双方约定指标。

## 5. 上板测试流程

### 5.1 准备工作

- 检查收发板电源、时钟、光纤及AD/DA连接。
- 确认发送端和接收端使用匹配的bitstream版本。
- 准备已经确认的VIO参数，包括PAM4门限、LMS系数和同步门限。

### 5.2 操作步骤

1. 给收发板上电。
2. 分别下载发送端和接收端bitstream。
3. 等待AD/DA初始化及GT lane训练完成。
4. 复位接收端业务同步模块。
5. 发送端开始循环发送固定训练序列。
6. 观察接收端互相关结果及采样窗口地址，确认训练同步完成。
7. 发送端从训练状态切换到PAM4测试数据或DES密文数据。
8. 检查接收端数据valid、PAM4判决和64 bit分组状态。
9. 启动SER、明文比较或CRC统计。
10. 记录错误数量、统计总量、运行时间及VIO配置。

## 6. 测试指标

### 6.1 无加密PAM4模式

至少记录：

- 业务同步是否成功。
- 互相关峰值位置及稳定性。
- PAM4三级判决门限。
- LMS系数。
- SER错误符号数和统计符号总数。
- 连续运行时间及重新同步次数。

当前工程每次SER统计16384个PAM4符号，实际符号错误率为：

```text
SER = 错误符号数 / 16384
```

### 6.2 DES模式

至少记录：

- DES密钥编号或测试密钥。
- 发送明文块数量。
- 恢复密文块数量。
- 解密正确块数量。
- 解密错误块数量。
- CRC错误帧数量。
- 连续运行时间及失锁次数。

## 7. 交付内容

### 7.1 验收前

- 收发端测试bitstream。
- 上板操作说明。
- VIO关键参数说明。
- 测试步骤及结果记录。

用户依据约定测试流程完成验收；测试结果达到双方约定指标后，确认验收并完成约定款项。

### 7.2 验收后

完成约定款项后，交付：

- 完整发送端Vivado工程。
- 完整接收端Vivado工程。
- DES加解密RTL源码。
- 测试平台及必要测试向量。
- 设计说明、接口说明和测试说明。

具体交付范围、器件版本、Vivado版本及验收指标以双方最终约定为准。

## 8. 风险与注意事项

- DES有效密钥长度为56 bit，仅适用于本项目实验或指定需求，不建议用于现代高安全等级系统。
- 收发端必须严格统一64 bit高低32 bit顺序和PAM4 bit映射顺序。
- DES流水线启动期间必须继续发送训练数据，待首个密文块准备完成后再切换到数据状态。
- DAC每个120 MHz时钟都需要完整的768 bit采样数据，不能出现无定义的valid空洞。
- PAM4门限、LMS系数或ROM地址错位都会导致SER升高，不应直接认定为DES算法错误。
- 第一版使用固定密钥，避免动态密钥流再次引入复杂的收发状态同步问题。
